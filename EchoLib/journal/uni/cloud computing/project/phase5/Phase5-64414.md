## Overview

The primary goal of Phase 5 is the migration of the microservices from a local containerized environment to a cloud-native deployment on **Google Kubernetes Engine (GKE)**. 

## Infrastructure Setup
The deployment is managed via Kubernetes manifests located in the project repository. These resources are deployed within a dedicated namespace to ensure isolation.

### Configuration and Secrets
- **ConfigMap (00-configmap.yaml)**: Centralizes non-sensitive environment variables such as database hostnames, ports, and gRPC URLs for inter-service communication.
- **Secret (01-secret.yaml)**: Stores sensitive data, including the PostgreSQL database URL, JWT secrets, and admin credentials, using opaque base64 encoding.

### Database Deployment
- **PostgreSQL**: Deployed as a single-replica deployment within the cluster.
- **Connectivity**: Uses a `ClusterIP` service to expose port 5432 internally to other microservices.
- **Health Checks**: Includes a readiness probe that executes pg_isready to ensure the database is fully initialized before accepting traffic.

## Microservices Deployment
Each microservice is deployed with specific resource requests and limits to optimize cloud costs while maintaining performance.

### Recommendation Service
- **Image**: *jrcarrasqueira/recommendations-phase5*.
- **Environment**: Pulls configuration from group8-config and sensitive URLs from group8-secret

### Subscriptions Service
- **Image**: *jrcarrasqueira/subscriptions-phase5*    
- **Integration**: Connects to the Users service via gRPC for verification.

3. Cluster for the project
```
gcloud container clusters create group8-cluster \ 
--zone europe-west1-b \ 
--num-nodes 2 \ 
--machine-type e2-standard-2 \ 
--disk-type pd-standard \ 
--disk-size 30 \ 
--enable-ip-alias \ 
--release-channel regular
```

### Applying Manifests

Run the following commands from the repository root:

Bash

```
# 1. Create Configs and Secrets
kubectl apply -f 00-configmap.yaml
kubectl apply -f 01-secret.yaml

# 2. Deploy Database
kubectl apply -f 02-postgres.yaml

# 3. Deploy Microservices
kubectl apply -f 10-recommendations-service.yaml
kubectl apply -f 11-subscriptions-service.yaml
```

### Verification

Check the status of the pods and services:

Bash

```
kubectl get pods
kubectl get services
kubectl get hpa
```

## 6. Accessing the APIs

Once deployed, the frontend services are accessible via the **Cloud Load Balancer** IP assigned to the Ingress or Gateway. Only necessary ports (REST) are exposed to the public internet, while gRPC remains internal to the cluster for security.
